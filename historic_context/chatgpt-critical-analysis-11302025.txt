## 1. Autopsy: what went wrong (and must never happen again)

### 1.1 Skipping DSIE and environment audits

You *explicitly* told them “Diagnose first,” but the build history shows the opposite:

* Multiple large “brain injection” and “v5.x” scripts were pushed before verifying basic conditions: IAM roles, Drive quotas, data integrity, or even whether Firestore documents contained real odds instead of error messages. 
* Only late in the game does the agent run a proper **ENVIRONMENTAL AUDIT (DSIE-00)** asking about service-account roles, key freshness, env vars on Kaji, org policies, and Drive vs GCS constraints. 

This is a direct violation of DSIE-01 (Diagnose), which demands a world-class structural, psychological, and data-integrity audit *before* implementation. 

> **Rule for your agents:**
> **No code, no scripts, no “let’s redeploy”** until a written DSIE-01 DiagnosisReport exists for the problem, with a falsifiable root-cause hypothesis and evidence from logs, Firestore, IAM, and the target site’s behavior.

---

### 1.2 Treating partial success as “architecture failure”

You actually *did* get a lot right:

* **Architecture:** Cloud Run orchestrator + Firestore mission queue + Dell/Tailscale worker is a sound “Cloud Brain + Residential Hands” design.  
* **Reality:** The Dell scraped OddsPortal successfully; the HTML clearly contains the odds and timestamps, plus your later screenshot pipeline works. The real failure was *parsing* noisy HTML, not the hybrid architecture itself. 

Instead of locking that in and hardening it, the prior engineer repeatedly suggested new big-bang scripts (v5.2, v5.6, etc.) when the right move was: “Keep v5, add a small validator and a better parser.”

> **Rule for your agents:**
> When something partially works (e.g., scraping succeeds, parsing fails), **freeze the working part as “known good”** and only operate on the failing stage. DSIE-02 literally tells you to invert the root cause, not rebuild the universe. 

---

### 1.3 Over-building & “God scripts”

The build history is full of **huge shell scripts** that:

* Create the entire repo from scratch (requirements, schemas, XML system prompt, DSIE core, FastAPI app, Dockerfile, deployment) in one shot. 
* Bake your entire Constitution + DSIE SOPs directly into one mega system prompt. 

Your own **System Instruction Manual** explicitly warns against:

* Relying on vague tools
* Overloading the context window
* Creating “God Agents” with giant prompts doing 10+ different tasks 

But your agents keep doing exactly that.

> **Rule for your agents:**
>
> * No script that **creates an entire application from scratch** unless you explicitly request a “fresh boot.”
> * Prefer **small, diff-style changes** against an existing repo. Summarize: “I will change these 3 files, in these 5 lines.”
> * No system prompt > ~2k words or > 3 distinct responsibilities per agent. If it grows beyond that, split into multiple agents.

---

### 1.4 Data integrity & storage mistakes

Two big recurring issues:

1. **Bad data treated as success**

   * Missions were marked COMPLETED and sent to Gemini even when Firestore stored only a generic error string instead of odds. 
   * This violates your own DSIE-01 **Data Integrity Audit** (“Quick Ping,” multi-source triangulation). 

2. **Drive quotas & service accounts**

   * The architect tried to let the service account *create* its own Drive folder, which ran straight into `storageQuotaExceeded` because service accounts have zero quota and can only write into a human-owned folder they’ve been granted access to. 
   * Only later do you hardcode the human-owned folder ID, which is the correct pattern.

> **Rule for your agents:**
>
> * Missions may not be marked COMPLETED until **domain-level data checks** pass (e.g., “this HTML has at least one odds row, at least one team name, and a plausible event date”).
> * Drive integration must always be: **user-owned folder ID + service-account writer role**, no “create folder if missing” logic in the SA context.

---

### 1.5 Governance gaps: DSIE-03/04 not wired into code

Your SOPs for Implement and Execute are excellent:

* DSIE-03: Gift-card pilot, sandbox, 50-point QA checklist, shadow run vs humans. 
* DSIE-04: Scoreboard lock, kill-switch protocol (“Profit or Delete”), autopsy loop. 

But in the code, there is **no actual scoreboard**, no execution reports, no automated kill-switch using real metrics; they exist only as Pydantic models and prose. 

> **Rule for your agents:**
> Every new “corporate subsystem” (e.g., odds harvester, lead generator, pricing engine) must launch with:
>
> * A **Primary Metric** (net profit, lead volume, etc.) and
> * An **ExecutionReport** JSON that the system actually writes somewhere you can see (Firestore/BigQuery dashboard). 

No metric, no deploy.

## 2. Design principles for the automated corporation

You already wrote 90% of the philosophy in:

* The **Constitution of Veiled Vector Inc.** 
* DSIE SOPs (Diagnose, Strategize, Implement, Execute) 
* The **Autonomous Agent Playbook & Instruction Manual**  

The gap is making them operational. Here’s how I’d distill them into *binding rules for your agents*.

### 2.1 Agents are employees with explicit authority

From your playbook: “Agents are employees, not assistants; system prompt = job contract.” 

For every agent, define:

* **Role & org chart:** e.g.,

  * *GCP_Architect_Agent* – designs infra, never touches money.
  * *Data_Acquisition_Agent* – owns Kaji worker & scraping logic.
  * *Profit_Engineer_Agent* – ties product behavior to revenue metrics.

* **Autonomy level:**

  * L0: read-only,
  * L1: can propose diffs,
  * L2: can apply low-risk infra changes in dev,
  * L3+: always requires human sign-off.

* **In-scope / out-of-scope:** what they absolutely **may not** do (e.g., create new GCP projects, rotate keys, or change billing without explicit approval).

This aligns directly with your manual’s “no God agents” and “authority boundaries”. 

---

### 2.2 Instruction architecture: smaller, sharper prompts

Your **System Instruction Manual** pushes a modular XML layout (`<meta>`, `<role>`, `<tools>`, `<constraints>`, `<examples>`). 

That’s perfect—but, in practice, your orchestrator prompt is trying to house:

* Constitution
* DSIE protocol
* Scratchpad rules
* Instruction hierarchy
* Sandwich defense
* Plus whatever task-specific logic gets added. 

Best practice for the corporation:

1. **Keep the Constitution & DSIE as shared “policy docs”, not fully inlined** into every agent. Let agents *fetch* the relevant article/section via RAG when needed, rather than carrying all of it in every system prompt.

2. **Per-agent system prompts** should be:

   * 1 role
   * 1 clear objective
   * A small set of tools
   * DSIE reference scoped to *their* responsibilities (“You must always perform DSIE-01 before changing infra.”)

3. Use **short, explicit tool descriptions**, as your manuals recommend: “tired intern–level detail” on inputs/outputs and failure handling. 

---

### 2.3 Governance & kill-switch, wired to real metrics

Your docs already define:

* The **Efficiency Covenant** (>10% gain vs manual) 
* The **Sandbag Protocol** (under-promise, over-deliver). 
* DSIE-03/04 QA and kill-switch mechanics.  

In the final corporation design, this should look like:

* **Central “Scoreboard” service** (BigQuery + simple dashboard) where every agent posts:

  * `agent_id, subsystem, primary_metric, actual_value, tokens_used, constraint_violations`
* A **Watcher/Governor agent** that:

  * Scans these metrics,
  * Marks agents as PROMOTE/PROBATION/TERMINATE per DSIE-04,
  * And opens “Autopsy Tasks” when things are terminated. 

That’s how you turn the Constitution from vibes into actual runtime behavior.

## 3. Concrete next steps & checklists (for your primary agents)

Let’s assume three flagship agents for the current GCP setup:

1. **GCP_Architect_Agent** – owns infra and deployment
2. **Kaji_Worker_Agent** – owns Dell/Tailscale + scraping reliability
3. **Profit_Engineer_Agent** – ensures everything ladders up to net profit

I’ll give each of them a tight, DSIE-compliant playbook.

---

### 3.1 GCP_Architect_Agent: from “epic scripts” to surgical diffs

**Mission:** Maintain a minimal, stable infra core on GCP; implement changes only after diagnosis; prefer diffs over rewrites.

**A. Before any change (DSIE-01 checklist)**

Agent must produce a written DiagnosisReport with:

1. **Structural audit**

   * Cloud Run revision & config (image, env vars, secrets).
   * Firestore collections & indexes relevant to the failing path.
   * Service account roles for `vector-archivist` (Firestore, Secret Manager, Cloud Run, Drive/GCS if used). 

2. **Data-integrity audit**

   * Sample Firestore docs for the failing missions (e.g., `mission_queue` entries) with key fields: `status`, `result_data`, `result_title`, `analysis_status`. 
   * Cloud Logging excerpts that show the actual error (Drive quota, safety blocks, parsing failures). 

3. **Environment audit**

   * Confirm current values and freshness of secrets (`GOOGLE_AI_STUDIO_KEY`, `TAILSCALE_AUTH_KEY`). 
   * Confirm org/project policies (public ingress allowed? SA key usage allowed?). 

Only once this report exists should the agent propose strategies (DSIE-02).

---

**B. How it should propose changes (DSIE-02 & 03)**

For each problem, the agent must output a **StrategyObject** with your three paths: Inversion, Stop-Loss, Asymmetric. 

Example pattern:

* **Inversion path:** e.g., “Implement Firestore → Eventarc → Cloud Run trigger to eliminate polling; keep current endpoints.” 
* **Stop-Loss path:** e.g., “Disable new missions for sports X until parsing improved; just log and drop.”
* **Asymmetric path:** e.g., “Use Gemini Vision on screenshots instead of parsing HTML, keeping existing hybrid architecture.” 

Changes must be expressed as:

* Repo-relative diffs (“In `src/dsie_core.py` I will add a data-integrity check around `result_data` and only call Gemini when length > 1k and title doesn’t contain ‘Error’."). 
* One shell command at most (e.g., `gcloud run deploy …`), not a whole new brain-injection script.

---

**C. Event-driven autonomy (v5.2 done right)**

Once manual `/process` calls are confirmed working, the architect agent should:

1. Design an **Eventarc or Firestore-triggered** flow:

   * On `mission_queue/{id}` where `status` transitions → `COMPLETED`,
   * Fire Eventarc → Cloud Run `/process` with `mission_id`. 

2. Implement it as:

   * A small Cloud Function or another Cloud Run service dedicated to listening to Firestore;
   * A config-only change (no re-writing your main orchestrator every time).

This is a one-time infrastructure win that directly enforces the Efficiency Covenant (no more manual polling). 

---

### 3.2 Kaji_Worker_Agent: make the Dell boring and immortal

**Mission:** Keep the Dell “hands” simple, stable, and always-on. No cleverness, just reliable work.

**A. Reliability**

* Run the worker as a **systemd service**, with:

  * `Restart=always`, `RestartSec=5`
  * Logging to journald + a lightweight local log file. 
* Ensure environment cleanliness:

  * No stray `GOOGLE_APPLICATION_CREDENTIALS` / proxy env vars that override what the worker expects. 

**B. Data quality gate at the edge**

For each mission:

1. After scraping, inspect:

   * Page title, key selectors, length of content.
   * Presence of domain-specific markers (team names, “Moneyline”, odds-like numerics).

2. If checks fail:

   * Mark mission as `status: FAILED_BAD_SOURCE` with `error_reason`.
   * **Do not** upload to Drive / set `status: COMPLETED`.

3. Only if checks pass:

   * Upload HTML and/or screenshot, update `status: COMPLETED`.

That way, the Cloud Brain never has to guess whether the input is garbage.

---

### 3.3 Profit_Engineer_Agent: tie everything to money

Right now, most of the work has been infra-heavy and product-light. The **Profit_Engineer_Agent** should exist to constantly ask:

> “Does this increase expected net profit or save time by at least 10%?” 

This agent’s job:

1. **Define product-level scoreboards** for each autonomous subsystem:

   * For odds harvesting: metrics like

     * `games_covered_per_day`,
     * `valid_odds_rows_per_game`,
     * token cost per processed game,
     * and downstream $ value if/when you plug into betting/trading logic.

2. **Enforce DSIE-03/04**:

   * For any new feature, require:

     * A **Gift-card pilot** (tiny, controlled run on a few games). 
     * A **QA score** (e.g., 45/50) before moving out of sandbox.
     * A **Kill-switch plan**: after 72 hours, if metric < sandbagged forecast, reduce its privileges or disable automatically. 

3. **Police build churn**:

   * Refuse to authorize yet another rebuild unless:

     * There is a clear DiagnosisReport showing why patching the current design cannot fix the issue, *and*
     * The expected ROI from the rebuild beats the Cost of Inaction (COI) per DSIE-02.
